## 3. 分析・考察（ルーブリック④ 15点）

システムの評価結果に対する分析と考察（実測最優先）。

- **うまくいった点**
  - 字幕変換遅延は `ja→en 0.76秒`, `ja→やさしい日本語 0.99秒` と目標 `<3秒` を達成し、リアルタイム支援として十分な応答性を確認。
  - 専門用語を含む5文で翻訳品質 `5/5 正確` を確認し、講義文脈での訳品質の安定性を示せた。
  - `sources==[]` 時の低信頼フォールバックを設計上強制し、根拠がないときに答えない挙動を維持できた。
  - `chunk_id` を持つソース構造により、回答根拠の追跡可能性を実装できた。

- **課題・限界**
  - QA E2E遅延は平均 `9.18秒` で目標 `<5秒` 未達。現環境ではLLMタイムアウトによりソース抜粋フォールバックが発生。
  - ASR補正は平均 `6.41秒`、補正採用率 `0/5`。Judge閾値が保守的で、誤認識修正の採択が厳しい。
  - ASR CERは自動再現計測が難しく、現時点は手動評価前提。
  - BM25フォールバックは開発運用上有効だが、耐久運用の主経路としては制約が残る。

## 4. プロンプト改善プロセス（ルーブリック② 20点）

AI活用の工夫として、初期設計の問題から改善後までを4段階で整理。

- **初期設計（初期プロンプトと発生した問題）**
  - QA回答プロンプトが実質日本語前提で、英語質問でも日本語回答が返るケースがあった。
  - 字幕変換でフォールバックが発生しても理由がUI上で不可視で、「翻訳が壊れた」ように見える問題があった。
  - タイトル生成系では出力が説明文寄りになり、短い候補語として使いづらい失敗があった。
  - 根拠不十分時の振る舞いを厳密に運用しないと、回答信頼性がぶれるリスクがあった。

- **改善後（改善内容と効果）**
  - 回答生成プロンプトを `Use only lecture sources` + 時刻引用 + 欠落時明示 + 言語指示付きに変更し、言語ミスマッチと外挿回答を低減。
  - Verifierプロンプトを「主張単位の支持判定(JSON)」に固定し、不支持主張の抽出→修正→再検証の流れを構築。
  - 字幕変換レスポンスを `status/fallback_reason` 付き構造化に変更し、フォールバック可視化で運用デバッグ性を改善。
  - タイトル生成は `<word>` 制約と LLM-as-a-judge（`OK/NG`）を導入し、出力品質を制御可能にした。
